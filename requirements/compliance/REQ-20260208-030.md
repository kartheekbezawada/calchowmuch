# REQ-20260208-030 — Implement Local SEO & Performance Audit Runner (Slug-Driven)

Requirement Type: Tooling / Compliance Automation
Status: NEW
FSM Phase: REQ
Depends On: REQ-20260208-029 (Audit Framework)
Applies To: All calculator categories and calculator pages in the REQ-20260208-029 slug list
Audit Target: Localhost only
Top Priority: SERP correctness (SEO > Performance)

---

## 0) Goal
Implement a deterministic, localhost-only audit runner that:

- Consumes an explicit slug list (no discovery)
- Audits each slug for SEO P0 items and required schema presence
- Runs Lighthouse (desktop + mobile) against localhost
- Writes compliance evidence to Git:
  - `requirements/compliance/Audit.md` (master table + per-page blocks)
  - raw Lighthouse JSON files under `test-results/seo/local-audit/...`

This REQ is the implementation companion to the framework defined in REQ-20260208-029.

---

## 1) Inputs & Configuration (MANDATORY)

### 1.1 Slug List (Driver)
Input slug list file:
- `requirements/compliance/REQ-20260208-029.slugs.txt`

Rules:
- Ignore blank lines and comment lines starting with `#`
- Normalize to leading `/` and trailing `/` when treating as a directory-style route
- Do NOT auto-discover additional routes; audit exactly the provided slugs

### 1.2 Base URL
Audit must run against:
- `BASE_URL=http://localhost:<PORT>`

Rules:
- No HTTPS required
- Do not depend on any production availability

---

## 2) Required Outputs (MANDATORY)

### 2.1 Audit Markdown (Human Log)
The runner MUST write/update:
- `requirements/compliance/Audit.md`

Format contract is defined in REQ-20260208-029 and MUST be followed exactly:
- Header block with BASE_URL + timestamp
- Master audit table containing EVERY slug
- Per-page detail blocks for EVERY slug

### 2.2 Machine Evidence (Raw Lighthouse JSON)
For each slug, capture:
- Desktop Lighthouse JSON
- Mobile Lighthouse JSON

Output path contract (must be deterministic):
- `test-results/seo/local-audit/<slugSafe>/lighthouse-desktop.json`
- `test-results/seo/local-audit/<slugSafe>/lighthouse-mobile.json`

Where `<slugSafe>` replaces `/` with `__` and trims leading/trailing separators.

### 2.3 Deterministic Pass/Fail Classification
Runner must classify each slug:

- **SEO Status:** PASS / WARN / FAIL
- **Schema Status:** PASS / FAIL / N/A
- **SERP Risk:** LOW / MED / HIGH

Mapping rules:
- Any SEO P0 violation => SEO Status = FAIL and SERP Risk = HIGH
- Any schema requirement failure on L3 calculator pages => Schema Status = FAIL and SERP Risk = HIGH
- Missing meta description or out-of-range description length => SEO Status = WARN (unless missing entirely, then FAIL)
- Lighthouse below thresholds => Performance is logged; SERP Risk increases to MED unless there is already HIGH

---

## 3) SEO Checks (Automated Minimum)

### 3.1 Indexability & Canonical (P0)
For each slug:
- Extract `<meta name="robots">` (if present) and ensure it contains `index` and `follow` and does NOT contain `noindex` / `nofollow`
- Extract canonical:
  - Must exist exactly once
  - Must be absolute
  - Must start with `https://calchowmuch.com/`

### 3.2 On-Page SEO (P0/P1)
- H1 count must be exactly 1 (P0)
- Title must exist; measure character length (WARN/FAIL thresholds per REQ-20260208-029)
- Meta description must exist and be 140–160 chars (WARN outside range; FAIL if missing)

### 3.3 Intent / Explanation Presence (P1)
Automatable checks (minimum evidence):
- Confirm page has non-trivial explanation content (text length threshold) OR at least one explanation heading element
- Record WARN if explanation appears empty/minimal

---

## 4) Schema Checks (Calculator Pages)

### 4.1 JSON-LD Extraction
- Extract all `script[type="application/ld+json"]` blocks
- Parse JSON (support both single object and array)

### 4.2 Required Types (P0)
For L3 calculator slugs, verify presence of:
- WebPage
- SoftwareApplication (EXACTLY ONE)
- FAQPage (page-scoped only)
- BreadcrumbList

FAQPage rules (P0):
- Exactly 10 questions

For non-calculator pages in the slug list:
- Schema Status may be `N/A`

---

## 5) Lighthouse (Local)

### 5.1 Execution
Run Lighthouse against `BASE_URL + slug` for:
- Desktop
- Mobile

Constraints (from WORKFLOW):
- If a user-data-dir is needed, it MUST be outside the repo (e.g. `/tmp/lighthouse-*`).
- Use Playwright’s Chromium path if needed (see WORKFLOW Lighthouse template).

### 5.2 Capture Metrics
From Lighthouse JSON capture:
- Performance score
- LCP
- CLS
- INP

Apply thresholds from REQ-20260208-029.

If Lighthouse cannot run in environment:
- Record Perf Score as `N/A`
- Mark page Perf result as WARN
- Keep SEO authoritative (do not convert SEO PASS into FAIL due to Lighthouse tooling failure)

---

## 6) Implementation Constraints

- Must be runnable on Linux in CI/local.
- Must not require a GUI.
- Must not require production domain resolution.
- Must not modify production site files as part of running the audit.

3) Audit evidence written to:
- `requirements/compliance/Audit.md`
- `test-results/seo/local-audit/**/lighthouse-*.json`
