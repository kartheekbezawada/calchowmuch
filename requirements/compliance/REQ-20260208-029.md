# REQ-20260208-029 — Local SEO & Performance Audit Framework (Calculators)

Requirement Type: Compliance / Audit Framework
Status: NEW
FSM Phase: REQ
Applies To: All calculator categories + calculator pages in scope (see Slug List)
Audit Target: Localhost only (no production dependency)
Top Priority: SERP correctness (SEO > Performance)

---

## 1) Purpose & Scope

### 1.1 Primary Objective
Create a repeatable, deterministic local audit that evaluates **SEO correctness + performance** for:

- L1: Category entry pages (top-nav entry points)
- L2: Calculator index pages (if present)
- L3: Individual calculator pages

Scope categories (at minimum):
- Finance
- Credit Cards
- Percentage Calculators
- Time & Date
- Home Loan
- Auto Loans

The audit MUST run against a local server and MUST persist results as compliance artefacts in Git.

### 1.2 What This Audit Is (and Is Not)
IS:
- Localhost, deterministic, slug-driven (no runtime discovery)
- SERP-focused: indexability, intent clarity, schema correctness, crawlable explanation content
- Captured as compliance evidence (Markdown + raw Lighthouse JSON)

IS NOT:
- Dependent on Google Search Console
- Dependent on live rankings
- A production crawl

## 4) SEO Audit Requirements (SERP-P0)

### 4.1 Per-Page SEO Checks (MANDATORY)
Each slug MUST be audited for the following.

#### A) Indexability & Canonical (P0)
- `<meta name="robots">` MUST contain `index,follow` (order/whitespace-insensitive)
- Canonical MUST:
  - exist
  - be absolute
  - reference production domain (`https://calchowmuch.com/...`) regardless of localhost base
- Duplicate canonicals: NOT allowed (P0)
- Any `noindex` (robots meta, x-robots-tag, or meta robots): NOT allowed (P0)

#### B) SERP Intent Alignment (P0/P1)
- Exactly one primary intent per page
- Calculator pages MUST be calculator-first (not article-first)
- Explanation pane SHOULD be present and crawlable (static HTML content, not empty shell), but detector misses are non-blocking for this audit.
- No mixed intents on a single slug

Automatable evidence (minimum):
- Detect presence of the calculator UI root and explanation container
- Detect at least one visible explanation heading (e.g., an H2 or section title) OR non-trivial text length

#### C) On-Page SEO (P0/P1)
- H1: exactly 1 (P0)
- H1 contains primary keyword (P1; record as WARN if subjective)
- Title optimal length: 35–61 chars (P1; ~580–610 pixels proxy; WARN outside range)
- Meta description optimal length: 110–165 chars (P1; mobile-safe floor ~110, desktop guidance up to ~165, ~920 pixels proxy; WARN outside range, FAIL if missing)
- Pixel width is authoritative in SERP snippets; character limits are an operational proxy for deterministic audits.
- Frontload critical keywords and value proposition early, because snippet truncation can hide trailing text.
- Google may rewrite titles/descriptions to match query intent; relevance and clarity are more important than exact character count.
- No ranking penalty is assigned solely for length overrun; over/under length is treated as CTR/snippet-display risk (WARN).
- Keyword stuffing: NOT allowed (P1; manual-only, must support Notes entry)

#### D) Structured Data (P0)
For L3 calculator pages, JSON-LD MUST include:
- `WebPage` (required)
- `SoftwareApplication` (required) — EXACTLY ONE per page
- `FAQPage` (required; page-scoped only, no global injection)
- `BreadcrumbList` (required)

Rules:
- JSON-LD must match visible content (P0). For FAQPage: visible FAQs must match Q/A text.
- FAQ count where FAQPage exists: minimum 4 and maximum 20 (`4–20`) (P0).

For L1/L2 pages:
- Schema Status can be `N/A` if they are not calculator pages.

## 5) Performance Audit Requirements (SERP-P1)

### 5.1 Lighthouse Metrics (Local)
Run Lighthouse against localhost.

Capture and log:
- Performance score
- LCP
- CLS
- INP

Thresholds:
- Performance score >= 80
- LCP <= 2.5s
- CLS <= 0.1
- INP <= 200ms

Rules:
- Performance is secondary to SEO.
- Any blocking performance failure MUST be logged and classified (P2) and must not overwrite SEO P0 failures.
- If Lighthouse cannot run in the environment, record Perf Score as `N/A` and set page Perf Status to `WARN` with an explicit note.

## 7) Audit.md Format (MANDATORY)

### 7.1 Header
`requirements/compliance/Audit.md` MUST start with:

- `# SEO & Performance Audit Log`
- Environment (localhost base URL)
- Audit Type: Pre-Deployment
- SERP Priority: P0
- Timestamp (ISO)
- Slug list file reference

### 7.2 Master Audit Table (MANDATORY)
Audit.md MUST include a master table with columns:

Category | Slug | SEO Status | Perf Score | Schema Status | SERP Risk | Notes

Rules:
- Every audited slug MUST appear
- SEO Status: PASS / WARN / FAIL only
- SERP Risk: LOW / MED / HIGH only
- Schema Status: PASS / FAIL / N/A

### 7.3 Per-Page Detail Block (REQUIRED)
Audit.md MUST include a block per slug:

- `## <slug>`
- SEO bullets (H1, Title length, Meta description, Canonical, Intent clarity)
- Schema bullets (WebPage, SoftwareApplication, FAQPage, BreadcrumbList)
- Performance numbers (score, LCP, CLS, INP)
- SERP Verdict: SAFE / WATCH / BLOCK (maps to LOW/MED/HIGH)
