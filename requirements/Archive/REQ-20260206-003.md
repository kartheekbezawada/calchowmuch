# REQ-20260206-003 — Align TESTING_RULES with package.json scripts

## Status: COMPLETE

- Type: Compliance / Documentation
- FSM Phase: COMPLETE
- Priority: HIGH
- SEO Impact: NO

## 1) Problem

Codex/implementers pause after BUILD and ask whether to continue to TEST because the repo’s testing governance documents reference commands that do not exist in `package.json`.

Examples:

- `requirements/universal-rules/UNIVERSAL_REQUIREMENTS.md` references `npm run test:unit`, `npm run test:seo:p1`, etc.
- `package.json` currently provides: `npm test`, `npm run test:e2e`, `npm run test:iss001`, `npm run lint`, `npm run format:check`, etc.

This mismatch makes “required tests” ambiguous during BUILD → TEST handoff.

## 2) Goal

Make test execution deterministic by ensuring:

1) Every test command referenced in canonical testing docs is runnable in this repo, OR the docs only reference commands that exist.
2) BUILD → TEST handoff is explicitly defined so implementers proceed without needing to ask “should I continue?”

## 3) Non-Goals

- Do not expand test scope or add new suites beyond what is needed for parity.
- Do not change the FSM phases or role separation.

## 4) Requirements (Acceptance Criteria)

### 4.1 Command Alignment

At least one of the following must be true after implementation:

#### Option A: Docs match scripts (preferred minimal change)

- Update `requirements/universal-rules/UNIVERSAL_REQUIREMENTS.md` so every command shown matches an existing `package.json` script.

#### Option B: Scripts match docs

- Add missing scripts to `package.json` so the documented commands exist and map to the correct underlying tooling.

### 4.2 Canonical Source

- The repository must have exactly one canonical “test governance” document that implementers follow for test selection and commands.
- If multiple testing docs remain, they must not conflict; one must explicitly defer to the other.

### 4.3 BUILD → TEST Handoff Behavior (Clarification)

- After BUILD passes, implementer proceeds to TEST using the required suites per the matrix, without pausing for confirmation.
- Exception: If E2E/ISS tests require a server and it is not running, implementer should start it (`npm run serve`) and then proceed.

## 5) Proposed Changes (Implementation Targets)

### 5.1 Update canonical testing doc

- File: `requirements/universal-rules/UNIVERSAL_REQUIREMENTS.md`
- Change: Replace non-existent command references with the actual scripts from `package.json`, or explicitly define the mapping.

### 5.2 Optional: Add missing scripts

- File: `package.json`
- Change: Add script aliases (e.g., `test:unit` → `vitest run`, `test:all` → run unit + e2e + iss) ONLY if you choose Option B.

### 5.3 Optional: Clarify workflow wording

- File: `requirements/universal-rules/UNIVERSAL_REQUIREMENTS.md`
- Change: Add one short statement in the TEST state description clarifying that implementer should proceed automatically to required tests once BUILD is PASS.

## 6) Evidence / Verification

- Show that the canonical doc’s listed commands run successfully (or at least exist and are correctly defined).
- Confirm the minimal required suite list for a “New calculator” change type is unambiguous.

## 7) Notes

This REQ is intended to reduce iteration churn and agent hesitation by removing ambiguity, not by forcing broader test execution.
